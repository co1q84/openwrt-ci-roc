# x86 与太乙定制设计

**目标**

只保留 `x86-64` 与 `太乙(jdcloud_re-cs-07)` 两个产品目标。路由器基础能力继续跟随主线，第三方应用仅保留 `HomeProxy`、`OpenClash`、`Lucky`、`Watchcat`、`banIP`、`Dufs`、`FileBrowser`、`Vlmcsd`、`iStore`，以及 `Docker`、`NAS` 相关入口。

**方案选择**

采用混合 rootfs 策略：

- `x86-64` 使用可写根文件系统镜像，避免 `squashfs + overlay` 累积占用空间后导致应用更新空间无法回收。
- `太乙` 保持稳定的 `squashfs sysupgrade` 升级路径，但预置独立应用数据区挂载逻辑，将 Docker、iStore、NAS、文件服务等持续写入数据迁移到外部数据分区，避免系统 overlay 被写满。

**配置策略**

- 将通用配置按“基础路由能力”和“第三方应用能力”进行精简，删除与当前产品无关的目标与服务。
- `x86-64` 与 `太乙` 分别保留独立目标配置，避免通过单个超大配置文件为多个产品叠加无关功能。
- 保留与路由器本体相关的网络、存储、管理功能，例如 LuCI 基础组件、终端、文件系统工具、磁盘管理、共享服务等。

**构建策略**

- 仅保留 `x86-64` 与 `JDCloud-NoWiFi-ImmortalWrt` 两个工作流。
- `x86-64` 工作流在下载与编译前移除针对 `squashfs` 的假设，切换到 ext4 可写根文件系统镜像。
- `太乙` 工作流继续构建 `squashfs` 固件，同时引入应用数据区初始化脚本。

**自定义脚本策略**

- `scripts/Roc-script.sh` 只保留当前产品需要的 feeds 和第三方仓库拉取逻辑。
- 删除 `PassWall/PassWall2`、`frp`、`wechatpush`、`oaf`、`aria2` 等不在需求内的第三方服务。
- 新增自定义 files 注入逻辑，为系统提供应用数据区初始化与目录迁移脚本。

**验证**

- 校验两个目标配置中只包含目标产品与指定第三方应用。
- 校验工作流仅剩两个构建入口。
- 校验脚本中不再拉取无关第三方软件包。
